Postgres CI/CD Automation

Postgres · Git · Octopus · Bamboo

I automated how Postgres schema and stored procedures moved from development through production: a GitHub repo held the authorized database state, a fetch step pulled the current picture from DEV, and we committed only the deltas approved for the next promotion. Bamboo validated and packaged the same migration bundle; Octopus applied it in order through QA, PRE-PROD, and PROD with gates and auditability.

Before automation, schema drift and hand-applied hotfixes made releases risky: it was hard to prove what had run where, and rollbacks were expensive. The program treated database artifacts like application code: versioned in Git, built in CI, and promoted with the same scripts in every environment.

The differentiator was change detection that compared declared tables and procedures against what was already deployed, emitting migration scripts only for real deltas. That kept releases small and reviewable, cut deployment failures by roughly 60%, and reduced manual database deployment effort by about 80% in engineer-hours per release.

End-to-end flow

From live DEV to Git as system of record, then the same migration path through every higher environment.

Git as source of truth. The repository stored DDL, versioned migrations, and procedure definitions representing what the pipeline was allowed to promote: not a casual dump of everything in DEV, but the authorized shape of the database for the release train.

Fetch from DEV. When schema or procedures changed in the development environment, a fetch script connected (with least-privilege, read-oriented access), introspected the catalog, and compared it to the repo. It produced candidate scripts or updated artifacts reflecting the real delta, reducing copy-paste and making “what changed” machine-generated and repeatable.

Curation for production movement. Not every experimental change in DEV shipped upward. The team identified the subset tied to approved work (pull requests, DBA review, release branches), so only promotable changes were committed. That governance step is what separated “DEV truth” from “production-bound truth.”

Same code, higher environments. After merge, Bamboo ran validation (syntax, policy checks, ordering). Octopus then executed the identical ordered migration set against QA, then PRE-PROD, then PROD, with environment-specific secrets, approvals, and backups. Each stage reused the same bundle, which is what made the path reproducible and auditable.

Dataflow

Information moves from Postgres DEV through capture and curation into Git, then through CI/CD into successively stricter databases.

The diagram emphasizes two ideas at once: capture from DEV is automated and comparable run over run, and promotion is the same artifact chain in QA, PRE-PROD, and PROD. Curation sits between them so experimental or out-of-scope DEV work does not ride along into production-bound commits.

What each piece did

Repository layout. Versioned migration files (and, where useful, baselines or snapshots) gave a clear ordering and history. Pull requests carried the human review that complemented automated diff output.

Fetch and change detection. Comparing declared tables and procedures to what upper environments already had avoided regenerating unchanged objects and kept pull requests about real deltas, directly supporting the ~60% drop in failure-prone releases.

Bamboo. Build steps enforced static checks and produced a deployable package Octopus could consume, so “what built” and “what ran” stayed aligned.

Octopus. Environment-specific variables and approvals separated automation from privilege: the same process everywhere, with stricter gates closer to PROD.

Governance and operations

Destructive or locking changes deserved explicit policy: separate scrutiny for drops and type changes, maintenance windows where needed, and documented forward-fix rollback patterns (Postgres often favors compensating migrations over reversing DDL in production). Secrets never lived in the repo; they flowed from Octopus and platform secret stores per environment.

In practice the team spent less time in SQL clients retyping scripts and more time on correctness, performance, and review quality. That is what the ~80% reduction in manual deployment hours was meant to unlock.
