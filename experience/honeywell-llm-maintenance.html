Honeywell · Backend Engineer · Jul 2022 to Nov 2024

LLM Based Maintenance Assist

Pgvector · Azure Document Intelligence · Azure Search

I designed a plug and play ingestion and retrieval platform for an LLM based maintenance assistant. Honeywell’s maintenance organizations span many industry domains, each with its own manuals, bulletins, and operational vocabulary, so the stack had to accept new corpora and filters without a ground up rebuild. The same core pipeline served that breadth: layout aware extraction, chunking aligned to procedures, embeddings in pgvector, and hybrid retrieval with Azure Cognitive Search, with small interfaces for parsers and embedders so models and backends could evolve independently.

Maintenance knowledge lived in PDFs, scans, legacy CMMS exports, vendor notices, and internal runbooks. The ingestion framework normalized those sources through Azure Document Intelligence, then applied chunking tuned to procedure boundaries and metadata that preserved document type, equipment identifiers, and domain or product line so retrieval could be scoped per business unit or site.

At query time, vector search over Postgres and semantic ranking in Azure Search were composed so answers drew on passages that matched technician intent, not only keywords: a clear step up from a lexical only baseline. The modular contract between ingestion, embedding, and search made it practical to trial new embedding models or routing strategies while continuing to ship features.

End to end workflow

From heterogeneous documents across Honeywell businesses to grounded answers: one shared platform, domain specific configuration at the edges.

Multi domain inputs

Different Honeywell segments reuse the same maintenance pattern (diagnose, procedure, parts, safety) but not the same document libraries. The platform treated each vertical as a tenant style corpus: connectors and metadata profiles pointed at the right document stores and tag vocabularies (equipment families, sites, regulatory context) while the extract → chunk → index → retrieve flow stayed identical.

Ingestion and indexing

Documents flowed through a shared extraction stage (layout and tables where it mattered for procedures), then normalization and chunking with stable identifiers and rich metadata for filtering. Chunks were written to pgvector for dense retrieval and surfaced through Azure Cognitive Search for semantic and hybrid ranking. Keeping those paths behind narrow interfaces meant a new domain could often be onboarded by adding parsers, taxonomy mappings, and index filters, not by forking the assistant.

Query path and grounding

A technician question triggered hybrid retrieval: embedding similarity plus search side semantic signals, constrained by domain and asset metadata when available. The top passages were passed to the LLM as context so responses stayed grounded in approved maintenance content, with room to tighten citations or policies per deployment.

What plug and play meant in practice

Parsers and extractors implemented a common output shape (text blocks, table cells, provenance) so new file types or vendor layouts plugged in without changing downstream chunking contracts. Embedders sat behind a thin API: the same job runner could target a new model dimension or provider once the store and index mappings were updated. Corpus configuration (which indexes, filters, and synonym lists applied) let another Honeywell business line stand up a maintenance assistant reusing the platform while keeping content isolated and relevant. That separation between shared mechanics and domain configuration is what made the system reusable across industries inside the company.
