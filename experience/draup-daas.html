Draup · SDE 2 · Dec 2024 to Present

DAAS

GraphQL · Airflow · Trino · Apache Spark · Multi-agent systems

I built and evolved a data as a service delivery framework: customer data requirements expressed as structured input drive scheduled pipelines that generate warehouse queries, execute them, and push results to the right destination. A GraphQL layer turns requirement JSON into SQL; Airflow orchestrates the full run path; Trino (and Spark where needed) executes the work. I then introduced a multi-agent stack to analyze requirements, pick the right physical tables and relationships, and iteratively generate and fix SQL or Spark code, improving typical query time by about 20% and cutting automated-run failures by about 40%.

Before this work, standing up a new customer export often meant bespoke jobs and fragile hand-written SQL. The framework treats a delivery as a contract: capture the requirement once, compile it to executable code, run it on a schedule, and surface clear lineage from configuration through execution. That pattern scales to many customers without multiplying operational toil.

Scheduled data delivery pipeline

From customer requirement metadata to compiled SQL and warehouse execution under Airflow.

Each delivery starts from the customer’s data requirement: which entities, filters, grains, and refresh cadence they need. The orchestration layer collects that configuration on the schedule we owe them, calls a GraphQL API that accepts the requirement as JSON, and receives back a SQL script tailored to our semantic graph and physical layout. Airflow sequences the steps (fetch requirements, invoke the API, materialize or stream results) so retries, alerting, and dependencies stay consistent across tenants.

Execution targets Trino against the analytical warehouse for interactive SQL paths; where volume or shape demands it, the same logical requirement can map to Spark jobs. Keeping compilation (graph API) separate from execution (engine) meant we could evolve the warehouse and customer contracts independently while preserving one operational playbook.

The diagram shows the linear spine operators rely on: everything funnels through Airflow so a failed compile or a slow cluster query shows up as the same class of incident, with logs tied to a requirement version and tenant. The optional agent step sits in the middle on purpose: it improves the artifact before it touches the warehouse, rather than patching failures only after the fact.

Multi-agent requirement analysis and code generation

Specialized agents for requirements, schema grounding, and iterative SQL or Spark codegen with engine feedback.

The enhanced framework uses a multi-agent system rather than a single-shot code generator. A requirement analyzing agent interprets the customer’s intent and constraints. A table and column selection agent grounds that intent in the warehouse: it proposes the relevant tables and columns, attaches human-readable descriptions, and surfaces relationships (join keys, cardinality) so generated code does not hallucinate foreign keys. A code generation agent then emits either SQL for Trino or Spark transformations, depending on the workload profile.

Generated code does not ship blindly. It runs iteratively on Spark or Trino; parse errors, semantic mismatches, and engine exceptions are fed back as structured feedback so the agents can revise predicates, joins, or projections. That closed loop is what drove the measured gains: shorter typical runtimes (about 20%) because plans were better constrained to real tables, and a much lower failure rate (about 40% relative reduction) because bad SQL was corrected before it burned a full delivery window.

Splitting responsibilities across agents keeps each step auditable: you can inspect which tables were selected and why before trusting a query plan. The feedback edge from the engine back into analysis and codegen is the critical difference from static templating: it encodes the warehouse’s real behavior, not only its documentation.

Impact summary

The framework turned customer exports into a repeatable product surface: GraphQL in, SQL out, Airflow in the middle, Trino and Spark at the bottom. Agent-assisted optimization and the multi-agent codegen loop delivered measurable improvements (about 20% faster typical query execution and about 40% fewer failures on automated runs) without abandoning the structured requirement contract that operations and customers already understood.
